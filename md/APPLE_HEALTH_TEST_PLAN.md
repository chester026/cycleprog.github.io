# Apple Health — тест-кейсы (Фаза 1 MVP)

> Тестировать только на физическом iPhone — HealthKit не отдаёт реальные данные в симуляторе без вручную засеянных сэмплов (см. APPLE_HEALTH_IMPLEMENTATION_PLAN.md §3.12).

---

## 1. Подключение — откуда и как

| # | Кейс | Шаги | Ожидаемый результат |
|---|------|------|----------------------|
| 1.1 | Путь к экрану | Profile → секция "Интеграции" → тап на "Apple Health" | Открывается AppleHealthScreen, показан disconnected-стейт с benefits-карточкой |
| 1.2 | Первое подключение | На AppleHealthScreen тап "Подключить Apple Health" | Открывается системный диалог iOS с запросом разрешений на все 6 типов (RHR, HRV, сон, вес, VO2max, активная энергия) |
| 1.3 | Разрешить всё | В диалоге iOS дать разрешение на все типы | Экран переходит в connected-стейт, все метрики начинают заполняться (может занять секунду — идёт fetchHealthSnapshot) |
| 1.4 | Разрешить частично | В диалоге iOS разрешить только часть типов (напр. только сон и вес) | Экран всё равно уходит в connected-стейт (HealthKit не сообщает, что часть отклонена); неразрешённые метрики показывают "Пока нет данных", приложение не крашится |
| 1.5 | Отклонить всё | В диалоге iOS отклонить все разрешения | `connect()` всё равно возвращает true (диалог был показан и закрыт) → экран уходит в connected, но ВСЕ метрики "Пока нет данных", recovery score = null. Стоит перепроверить, не выглядит ли это как баг для пользователя — возможно в Фазе 2 нужен явный "нет данных" стейт вместо "connected с пустыми полями" |
| 1.6 | Android-устройство | Открыть AppleHealthScreen на Android (если есть тестовое устройство/сборка) | Показывается блок "Apple Health доступен только на iOS", кнопки подключения нет |
| 1.7 | Повторный запуск диалога | Отключить доступ в iOS Settings → Privacy → Health → BikeLab, затем в приложении тапнуть "Подключить" снова | iOS должен снова показать диалог разрешений (или сразу открыть Settings, в зависимости от версии iOS) |
| 1.8 | Персистентность после рестарта | Подключить Health → полностью закрыть приложение (не просто в фон) → открыть снова | Экран сразу в connected-стейте без повторного запроса разрешений (используется кэш из AsyncStorage, см. §5) |

## 2. Отключение

| # | Кейс | Шаги | Ожидаемый результат |
|---|------|------|----------------------|
| 2.1 | Отключение в приложении | На connected-экране тап "Отключить Apple Health" → подтвердить в алерте | Алерт с текстом про то, что это только очищает кэш, реальный доступ отзывается через iOS Settings. После подтверждения — экран уходит в disconnected-стейт |
| 2.2 | Отмена отключения | Тап "Отключить" → в алерте нажать "Отмена" | Ничего не меняется, экран остаётся connected |
| 2.3 | Отключение не трогает реальный грант | Отключить в приложении → перепроверить iOS Settings → Privacy → Health → BikeLab | Разрешения в iOS Settings остаются как были (приложение не может их отозвать программно) |
| 2.4 | Повторное подключение после отключения | Отключить в приложении → тап "Подключить" снова | Поскольку разрешение в HealthKit уже выдано раньше, `requestAuthorization` может не показать диалог повторно (это нормально) — снапшот должен обновиться и данные подтянуться |

## 3. Вызов/получение данных Health

| # | Кейс | Шаги | Ожидаемый результат |
|---|------|------|----------------------|
| 3.1 | Есть Apple Watch, полные данные | Подключить на устройстве с историей сна/HRV/RHR за 14+ дней | Все метрики заполнены реальными числами, recovery score рассчитан |
| 3.2 | Нет Apple Watch (нет HRV) | Подключить на iPhone без Watch (только вес/сон, если есть) | HRV = "Пока нет данных", но recovery score всё равно считается по формуле переnormalization (сон + RHR, без HRV) — не должен быть null, если есть хотя бы сон |
| 3.3 | Нет данных о сне вообще | Подключить на устройстве без единой записи сна за последние сутки | Recovery score = null (сон — обязательный сигнал), остальные метрики показываются, если есть |
| 3.4 | Только частичный сон (без deep-сна) | Есть asleepCore, но нет asleepDeep за ночь | sleepHours считается, но deepSleepPct = null или 0%; recovery score считается без sleep-quality компоненты |
| 3.5 | Вес в других единицах | Сменить единицы измерения устройства на фунты в Здоровье/Health → проверить AppleHealthScreen | Вес всё равно показывается в кг (unit форсирован в 'kg' в healthService.ts), независимо от локали устройства |
| 3.6 | Обновление вручную | На connected-экране тап "Обновить данные" | Крутится спиннер, снапшот перезапрашивается заново (fetchHealthSnapshot), метрики обновляются |
| 3.7 | Устаревший кэш (>4ч) | Подключить → подождать (или подменить timestamp в кэше) → переоткрыть экран/приложение спустя 4+ часа | При маунте useHealthData тихо обновляет снапшот в фоне (без спиннера на весь экран, если кэш уже был показан) |
| 3.8 | Свежий кэш (<4ч) | Открыть/переоткрыть приложение сразу после подключения | Данные показываются мгновенно из кэша, HealthKit не дёргается повторно (можно проверить логами — не должно быть новых вызовов queryQuantitySamples) |
| 3.9 | Ошибка HealthKit-запроса | Сложно воспроизвести искусственно — но можно временно сделать identifier невалидным и проверить | Функции fetchHealthSnapshot/latestQuantity не крашат приложение, просто возвращают null для этого поля (все fetch-хелперы обёрнуты в try/catch) |
| 3.10 | AsyncStorage недоступен | Отключить доступ к хранилищу (редкий кейс, можно пропустить если нет прямого способа) | getCachedHealth/disconnectHealth не бросают исключение, просто console.warn |

## 4. Интеграция с чатом коуча (health_context)

| # | Кейс | Шаги | Ожидаемый результат |
|---|------|------|----------------------|
| 4.1 | Health подключён, вопрос про готовность | В чате коуча спросить "Стоит ли мне сегодня делать интервалы?" | Коуч ссылается на конкретные цифры (recovery score, сон, HRV/RHR) в ответе |
| 4.2 | Health подключён, вопрос не про здоровье | Спросить про выбор велосипеда/питание | Коуч НЕ упоминает данные сна/HRV — это нерелевантно |
| 4.3 | Health НЕ подключён, вопрос про готовность | В чате спросить "Я восстановился после вчерашней тренировки?" | Коуч кратко предлагает подключить Apple Health (Profile → Apple Health), не повторяет это на каждый следующий вопрос |
| 4.4 | Health НЕ подключён, вопрос не про здоровье | Спросить про план тренировок на неделю | Коуч НЕ предлагает подключить Health — не в тему |
| 4.5 | health_context не логируется | Временно включить лог health_context на сервере (см. чеклист в implementation plan) → отправить сообщение с подключённым Health → проверить логи | health_context виден только во временном debug-логе, который нужно убрать перед мёржем; в постоянных логах сервера (console.log запроса) его быть не должно |
| 4.6 | health_context не сохраняется в БД | Отправить несколько сообщений с подключённым Health → сделать `SELECT * FROM coach_messages WHERE conversation_id = ...` | Столбец content/tool_calls/suggestions не содержит health_context ни в каком виде |
| 4.7 | Первое сообщение в новом чате | Начать новый чат сразу после подключения Health, спросить про готовность | Здоровье уже должно быть в контексте первого сообщения (healthContext из useHealthData доступен сразу после подключения, не только со второго сообщения) |

## 5. UI / состояния экрана

| # | Кейс | Шаги | Ожидаемый результат |
|---|------|------|----------------------|
| 5.1 | Загрузка при первом маунте | Холодный старт приложения, сразу открыть AppleHealthScreen | Показывается спиннер до того, как useHealthData прочитает кэш/статус |
| 5.2 | "Пока нет данных" | Подключено, но конкретная метрика ещё не пришла | Строка метрики показывает "Пока нет данных" вместо пустоты/крэша |
| 5.3 | Спиннер на кнопке Connect | Тап "Подключить" | Кнопка показывает спиннер вместо текста на время запроса |
| 5.4 | Спиннер на кнопке Refresh | Тап "Обновить данные" | Кнопка показывает спиннер, остальной экран не блокируется |
| 5.5 | Privacy-текст виден | Оба состояния (connected/disconnected) | Текст про "данные остаются на телефоне" виден в обоих случаях |
