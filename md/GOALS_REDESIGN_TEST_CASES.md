# Тест-кейсы редизайна системы целей

Автоматизированные (unit) кейсы для чистой логики — `server/test_goalCalculator.js`
(`node server/test_goalCalculator.js`, 33/33 зелёных). Ниже — сценарии, которые
нельзя проверить без реальной БД/OpenAI/устройства: ручные QA-кейсы по каждому
из трёх решённых вопросов + regression-кейсы на legacy-цели.

---

## 1. Создание цели с произвольной метрикой (проблема "захардкожены типы")

| # | Сценарий | Шаги | Ожидаемо |
|---|---|---|---|
| 1.1 | Цель на восстановление (`source: coach`) | В чате: "Поставь цель — 3 раза в неделю делать recovery ride следующие 2 недели" | `create_goal` создаёт sub-goal с `metric: {source:'coach'}`, `unit` осмысленный (напр. "rides"), `start_date`/`end_date` = 14 дней. Прогресс двигается только коучем через `update_goal`, не автоматически. |
| 1.2 | Цель на скилл (Climbing) | "Хочу подняться в скилле Climbing до 80" | Sub-goal с `metric: {source:'skills', skill:'climbing'}`, `target_value: 80`. `GET /api/goals` возвращает `current_value` = текущее значение `climbing` из последнего `skills_history` снапшота. |
| 1.3 | Цель на health-метрику (HRV) | "Хочу держать HRV выше 50" | Sub-goal `metric: {source:'health', health_metric:'hrv'}`. **Важно**: `current_value` с сервера — просто последнее сохранённое значение (не пересчитано). Живое значение показывается только в приложении через `useHealthData()`. |
| 1.4 | Цель на активность с фильтром | "Хочу сделать 5 покатушек с набором больше 1000м за месяц" | `metric: {source:'activity', aggregate:'count', filter:{min_elevation:1000}}` (или `min_elevation_rate`), `start_date`/`end_date` = месяц от текущей даты. |
| 1.5 | Невалидная метрика от модели (regression) | Смоделировать ответ AI с `metric: {source:'activity', aggregate:'total'}` (несуществующий aggregate) | `validateMetric` отклоняет — `aiGoals.js` бросает `Invalid sub-goal structure`, цель не создаётся молча с мусором. |

## 2. Произвольные периоды (проблема "захардкожены периоды")

| # | Сценарий | Шаги | Ожидаемо |
|---|---|---|---|
| 2.1 | Цель на 1 неделю | "Цель на эту неделю — 150км" | `start_date` = сегодня, `end_date` = +7 дней. `GoalDetailsScreen` показывает диапазон дат вместо "4 недели". |
| 2.2 | Цель на нестандартный срок (10 недель) | "Хочу подготовиться к гонке через 10 недель" | `end_date` = сегодня + 70 дней, а не округление к ближайшему из `4w/3m/year`. |
| 2.3 | "Сегодня" не устаревшая дата (regression) | Проверить промпт `aiGoals.js` — сравнить `${todayISO}` в сгенерированном JSON с реальной текущей датой сервера | Дата в `start_date` новой цели = дата запроса, не дата обучения модели и не зафиксированная в промпте константа. |
| 2.4 | Pace-бейдж | Создать цель с `end_date` в будущем, current_value искусственно занизить в БД | `get_goals_progress`/`GET /api/meta-goals/:id` возвращают `pace.onTrack: false`, `percentDelta < 0`; в `GoalDetailsScreen` — красный бейдж "Есть отставание". |
| 2.5 | Pace = null для legacy-цели | Открыть старую цель (создана до редизайна, `period: '4w'`, нет `start_date`) | `pace` = `null`, бейдж не рендерится (нет ложного "on track"/"behind"). |

## 3. Депрекейт библиотеки тренировок (проблема "ограниченный список тренировок")

| # | Сценарий | Шаги | Ожидаемо |
|---|---|---|---|
| 3.1 | Новая цель, вкладка Trainings | Открыть `GoalDetailsScreen` новой (пост-редизайн) цели → таб "Trainings" | Вместо старого списка тренировок — кнопка "Спросить план у коуча". |
| 3.2 | CTA переходит в чат с prefilled-промптом | Нажать кнопку из 3.1 | Открывается `CoachChat` с `initialPrompt`, содержащим название цели; сообщение отправляется автоматически (как и остальные `initialPrompt`-переходы в приложении). |
| 3.3 | Legacy-цель сохраняет старое поведение (regression) | Открыть цель, созданную до редизайна (есть `trainingTypes` в `ai_context`) | Вкладка Trainings показывает старые AI-generated карточки тренировок как раньше — библиотека не удалена, только не используется для новых целей. |

## 4. Авто-комплит на 98% (коуч-опосредованный, не молчаливый)

| # | Сценарий | Шаги | Ожидаемо |
|---|---|---|---|
| 4.1 | 98% — коуч поднимает тему | Довести все sub-goals меты-цели до ≥98% (напр. 393/400км), спросить коуча "как дела с целью?" | Коуч упоминает `readyToComplete` и **спрашивает**, закрыть ли цель — не заявляет её выполненной сам. |
| 4.2 | Пользователь соглашается → реальное закрытие | Ответить "да, закрывай" | Коуч вызывает `update_goal({goal_id, status:'completed'})`; `meta_goals.status` меняется в БД только после этого вызова. |
| 4.3 | Пользователь отказывается → статус не меняется | Ответить "нет, хочу продолжить" | `update_goal` не вызывается, `status` остаётся `active`, `readyToComplete` продолжит приходить `true` на следующих запросах. |
| 4.4 | 97% — флаг ещё не поднимается | Sub-goal на 97% от target | `readyToComplete: false` — порог строго 98%, не округление до "почти готово". |
| 4.5 | Overachieving (>130%) | Sub-goal на 135% от target | `overachieving: true`; коуч предлагает поднять `target_value` через `update_goal`, а не тихо оставляет цель "перевыполненной". |
| 4.6 | Expired | `meta_goals.target_date` в прошлом, `status: 'active'` | `expired: true`; коуч спрашивает — продлить/скорректировать/закрыть. Не автозакрывается. |

## 5. Health-source: клиент vs сервер (архитектурный инвариант)

| # | Сценарий | Шаги | Ожидаемо |
|---|---|---|---|
| 5.1 | Сервер никогда не считает health сам | Дёрнуть `GET /api/goals` для юзера с health-source sub-goal, не открывая приложение (нет свежего HealthKit-снапшота) | `current_value` в ответе — последнее сохранённое значение, не пересчитанное «в фоне» (сервер физически не может достать HealthKit). |
| 5.2 | Клиент подменяет значение живым | Открыть `GoalDetailsScreen`/`MetaGoalCard` с подключённым Apple Health | Отображаемый `current` берётся из `useHealthData().healthContext` через `getHealthMetricValue()`, а не из API-ответа. |
| 5.3 | Apple Health не подключен | Открыть ту же карточку без HealthKit-доступа | `getHealthMetricValue` возвращает `fallback` (последний `current_value` с сервера), не 0 и не крашится. |
| 5.4 | Health-данные не попадают в Postgres (инвариант) | Проверить код: нет ни одного `INSERT`/`UPDATE`, пишущего `hrv`/`resting_hr`/`sleep`/`weight` в `goals.current_value` со стороны сервера | Подтверждено — `goalCalculator.js` для `health` source только читает `current_value`, никогда не вычисляет и не пишет. |

## 6. Обратная совместимость (legacy-цели)

| # | Сценарий | Шаги | Ожидаемо |
|---|---|---|---|
| 6.1 | Старая цель без `metric` | Открыть цель, созданную до миграции (`metric IS NULL`, есть `goal_type`/`period`) | `GET /api/goals` считает прогресс через `legacyCalculator` (старый switch/case), не роняет запрос. |
| 6.2 | `avg_hr_hills`/`avg_hr_flat`/`intervals` (сознательно не автосчитаются) | Открыть такую legacy-цель | Прогресс остаётся как и раньше (0 для intervals, физическая модель для hr_hills/flat) — редизайн их не трогает и не должен. |
| 6.3 | Смешанная мета-цель (legacy + новый sub-goal) | Мета-цель с одним sub-goal на `goal_type` и одним на `metric` | `MetaGoalCard`'ный средний % считает оба корректно — фильтруется только `ftp_vo2max`, остальное усредняется одинаково независимо от источника. |

---

**Что не покрыто и требует ручной проверки на реальном стенде** (нет доступа к живому Postgres/OpenAI/iOS-устройству из песочницы): полный SSE-цикл `create_goal`→ответ модели→запись в БД; реальный HealthKit snapshot; фактическая рендер-проверка pace-бейджа и CTA-кнопки на устройстве.
